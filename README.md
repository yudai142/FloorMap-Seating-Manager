# FloorMap Seating Manager

## 1. サービス概要

部屋の上面図をブラウザ上で自由に描き、座席を配置して着席状況をリアルタイムで管理できるWebアプリ。
「入室前に誰がどこにいるか」を一目で把握でき、勉強会・会議・教室はもちろん、飲食店・ホテルロビー・イベント会場など様々な場面の座席管理を直感操作で実現する。
紙の座席表・ホワイトボード・口頭確認といったアナログ運用を、シンプルなビジュアル操作に置き換える。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

Runteqの受講生グループでセルフカフェを使ったもくもく会を開催していたとき、毎回「みんながどのあたりに座っているか」が入室前にわからず、店内で探し回る場面が何度もあった。
ある日、グループとは関係ない別のお客さんに声をかけてしまい、気まずい思いをした。その瞬間、「入室前にみんながどこにいるか確認できれば、こんな思いをしなくて済むのに」と強く感じた。

また、飲食店やカフェのホール業務でも同様の課題があることを知った。混雑時に「どのテーブルが空いているか」「あのお客様はどこにいるか」をスタッフ間で共有するのに口頭確認やホワイトボードへの手書きで対応しており、ピーク時に情報が食い違い、案内ミスや二重案内が発生するケースがある。「誰がどこにいるかをリアルタイムで共有できない」という課題は、カジュアルな集まりから業務の現場まで、場面を問わず共通して存在していると気づいた。

### 2-2. なぜそれが「気になった」のか

以前に入室・退室の記録ができる出席管理アプリを自作した経験があり、「人の所在を管理する仕組み」への関心がもともと高かった。
その経験から「記録できること」の価値は知っていたが、もくもく会での体験で「直感的に・ビジュアルで・リアルタイムに」所在を共有できることの重要性を改めて実感した。
また、勉強会という特定の用途にとどまらず、業務現場・会議室・教室・イベント会場など幅広い場面で使える汎用的な仕組みとして形にしたいと思ったことも動機のひとつ。
「仕組みで解決できることは仕組みで解決すべき」という価値観から、同じ情報を何度も人が確認・共有する手間を減らしたいという思いが強い。

## 3. 課題の整理

### 3-1. 表に見えている困りごと

もくもく会などのグループ集まりで入室前にメンバーの席がわからない、また飲食店・イベント会場などの現場でスタッフ間の座席情報共有が追いつかない。

### 3-2. 本当に解決したい課題は何か

**【カジュアルな集まり】**
- **なぜ困る？** → 入室しても誰がどこにいるかわからず、探し回ったり関係ない人に声をかけてしまう
- **なぜ困る？** → 気まずい思いをしたり、合流までの無駄な時間・ストレスが発生する
- **なぜ困る？** → 「自分がグループに合流できているか」という不安が、その場への参加ハードルを上げてしまう

**【業務現場】**
- **なぜ困る？** → 空席を即座に確認できないので、お客様をすぐ案内できない
- **なぜ困る？** → 案内が遅れると顧客満足度が下がり、スタッフ間の情報食い違いで二重案内・ミスが発生する
- **なぜ困る？** → スタッフの精神的負荷と業務ミスが増え、サービス品質全体が低下する

→ **本当に解決したい課題：所在情報がリアルタイムで共有されないことで、グループ活動・業務現場を問わず、余計なストレス・ミス・非効率が生まれること**

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

**年齢設定**
- 年齢：20〜40代。勉強会・もくもく会の参加者から、飲食店・ホテル・イベント会場のホール担当スタッフ・運営責任者まで
- 理由：スマートフォン・タブレットを業務や日常で使いこなす世代であり、新しいWebツールへの抵抗感が低く、ツール導入の意思決定にも関わりやすい

**生活・状況設定**
- 状況①（カジュアル）：週1〜複数回、カフェや共用スペースでグループもくもく会・勉強会・チームミーティングを開催している
- 状況②（業務）：飲食店・ホテルロビー・コワーキングスペース・イベント会場などで座席・テーブル管理を担当している
- 理由：どちらも「自由席の空間で、誰がどこにいるかが常に変わる」という共通の課題構造を持っており、同じツールで解決できる

**環境設定**
- 環境：参加者・スタッフが各自スマートフォンやタブレットを持ち、入室前・移動中・業務中にもアクセスできる
- 理由：「入室前に確認する」「フロアを歩きながら確認する」という使い方が核心であり、インストール不要のWeb完結型であることが重要

### 4-2. 自分とユーザーの距離

**過去の自分・現在の自分（カジュアル用途）**
Runteqのもくもく会で実際に気まずい体験をした当事者であり、課題の解像度が非常に高い。また出席管理アプリの開発経験があるため、「何が足りなかったか」を技術的にも体験的にも理解している。

**身近な誰か（業務用途）**
飲食・サービス業に携わる知人・友人の現場経験から課題を観察・共感しており、口頭確認やホワイトボード運用の非効率さへの理解がある。

### 4-3. 実際に使われる可能性について

- **最初に使ってくれそうな人：** Runteqの同期・もくもく会の主催者、および小規模飲食店・カフェのオーナーや店長
- **思い出す場面（カジュアル）：** もくもく会の開始直前、「今日みんなどこにいる？」とSlackで聞こうとした瞬間
- **思い出す場面（業務）：** 混雑ピーク時に案内ミスが起きた直後、または新店舗立ち上げ時のオペレーション設計中
- **今の代替手段：** Slackで「どこにいる？」と聞く・目視で探す（カジュアル）／ホワイトボード・紙の座席表・口頭確認（業務）
- **置き換える理由：** リアルタイム同期・複数端末での閲覧・着席履歴の記録という、アナログ手段では実現できない価値がある。操作は「席をクリックして名前を入力するだけ」で、代替手段より明らかに速くストレスがない

### 4-4. 導入・継続の流れ

**【カジュアル用途】**
- きっかけ：もくもく会の主催者が「毎回どこにいるか確認が大変」と感じたタイミングで導入
- 最初の一歩：よく使うカフェのレイアウトを部屋エディタで作成 → フロアページのURLをグループのSlack・LINEに共有
- 使い続ける理由：「集まる前にURLを開いて席を登録する」という習慣が生まれ、使わないと逆に不便になる

**【業務用途】**
- きっかけ：繁忙期の案内ミス・スタッフ間の情報食い違いが続いたタイミングで「何かツールはないか」と検索
- 最初の一歩：自店のフロア図を部屋エディタで作成（5〜10分）→ フロアページURLをスタッフのスマホに共有するだけで即日運用開始
- 使い続ける理由：毎営業日の着席管理に使うことで習慣化。着席履歴データが蓄積し、混雑時間帯の分析や座席レイアウト改善の判断材料にもなる

**toBの観点（業務導入時）：**
- 初期設定5分・インストール不要で導入コストが極めて低い
- 顧客の個人情報を扱わず、来客名（任意）と着席時刻のみ記録するためデータセキュリティのハードルが低い
- 既存のPOSや予約システムとは競合せず「フロアの今を可視化する補完ツール」として乗り換えではなく追加導入しやすい

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

- **サービス名：** TableCheck（テーブルチェック）
- **URL：** https://www.tablecheck.com/ja/
- **できること：** 飲食店向け予約・顧客管理・フロアマップ管理。テーブルの着席状況をビジュアルで管理できる

### 5-2. それでも作りたい理由

TableCheckは飲食店の予約・CRM管理を主目的とした有料SaaSで、「仲間内のもくもく会」「社内の会議室」「学校の教室」といったカジュアルな用途には機能過多でコストも見合わない。
また業務用途でも「今日だけ使いたいレイアウトを5分で作ってすぐ共有する」という即興性・手軽さが既存サービスにはなく、カジュアルな集まりから業務現場まで横断して使える汎用性も持っていない。そこへの不満と「自分で作れる」という確信が作りたい理由の核心。

### 5-3. 差別化を一文で決める

**「勉強会・会議・飲食店・イベント会場など、あらゆる集まりの場で入室前・案内前に全員の居場所をすぐ把握したい人」の「合流・案内の直前の確認の瞬間」を一番助けるアプリ**

## 6. 提供したい価値

### 6-1. ユーザーの変化

- **行動の変化：** 「今どこにいる？」をSlackで聞いたり探し回ること、スタッフ間での口頭確認やホワイトボード更新がなくなり、URLを開けば全員の席が即座にわかるようになる
- **気持ちの変化：** 知らない人に声をかけてしまう気まずさ・合流できるか不安なまま入室するストレス・案内ミスへのプレッシャーがなくなる
- **考え方の変化：** 「場の情報は共有ツールで管理するもの」という意識が生まれ、集まりの運営・現場オペレーションをより仕組み化して考えるようになる

### 6-2. 価値を一文で表す

「このサービスは**グループの集まりや座席管理が必要な現場で働くすべての人**が、**誰がどこにいるかをリアルタイムで一目把握**できるようになるサービス」

## 7. このアプリで実現すること

### MVPで作る機能

- 部屋の新規作成（名前・幅・高さ）
- SVGキャンバスで壁（線分）・座席を描画・配置・保存
- フロアページでの着席・離席・席移動
- ActionCableによるリアルタイム同期（複数端末対応）

### 本リリースで作る機能

- 着席履歴の一覧・CSV出力
- 複数フロア（部屋）の切り替え表示
- 座席ごとのメモ・タグ設定（禁煙席・VIP席・スタッフ専用など）
- スマートフォン向けUIの最適化
- 認証機能（グループ・店舗・組織ごとのアカウント管理）
- 着席時間のタイマー表示・長時間着席アラート
- よく使うレイアウトのテンプレート機能（カフェ・会議室・教室・ホテルロビーなど）

## 8. 懸念点とその対策

### 懸念点① 自分から使い始めてもらえるか（初期利用のハードル）

- **問題：** グループ全員・スタッフ全員が使わないと価値が出ないサービスのため、主催者・管理者が導入しても参加者・スタッフが着席登録をしてくれない可能性がある
- **理由：** 新しいツールへの登録・操作を面倒に感じる人がいると、所在情報が不完全になりアプリの価値が半減する
- **対策：** アカウント登録不要でURLを開くだけで着席できる設計にする。操作は「席をクリックして名前を入力するだけ」にとどめ、参加者・スタッフの導入コストを極限まで下げる

### 懸念点② SVGエディタのモバイル対応

- **問題：** SVGのドラッグ操作がタッチデバイスで直感的に動かない可能性がある
- **理由：** タッチイベントとマウスイベントは別途対応が必要で、実装コストが高い
- **対策：** MVPではPCブラウザでのエディタ操作を前提とし、フロアページ（着席管理）のみスマートフォン対応を優先する

### 懸念点③ リアルタイム同期の接続安定性

- **問題：** ActionCableのWebSocket接続が不安定な環境（公衆Wi-Fi・モバイル回線）では同期が遅延・切断する可能性がある
- **理由：** カフェ・店舗現場ではネットワーク環境が安定していないケースが多い
- **対策：** 接続切断時に自動再接続する処理を実装し、再接続後に最新状態をREST APIでフェッチして画面を復元する設計にする

## 9. 今後の展開・発展の方向性

### 観点

- **使い続ける理由：** 毎回の集まり・毎営業日の管理で「URLを開いて席を登録する」が習慣化されると、使わないと逆に不便になる
- **データが溜まると：** よく使われる座席・時間帯・混雑傾向が可視化でき、レイアウト改善やシフト設計の判断材料になる
- **他の人との関わり：** QRコードを印刷して壁・テーブルに貼るだけで来た人が自分でチェックインできるセルフ着席機能
- **運用での価値：** テンプレートを共有・公開できるコミュニティ機能（Runteqコミュニティ・飲食店オーナーコミュニティでの共有など）

### 今考えている発展案

- **機能追加：** セルフチェックイン（QRコード）・分析ダッシュボード・Slack通知連携（「〇〇さんが着席しました」）・予約システム連携（予約済み座席の事前色分け表示）
- **UI改善：** フロアページをタッチスクリーン対応の大型表示モードに対応（店舗のタブレット常設運用を想定）
- **機能以外の工夫：** カフェ・会議室・教室・ホテルロビーのレイアウトテンプレートを初期提供し、0から描かなくてもすぐ使い始められる体験を作る

## 10. 技術スタック

### 10-1. 使用予定の技術

- **フレームワーク：** Ruby on Rails 8 / React 19 (TypeScript) / Inertia.js
- **DB：** PostgreSQL
- **リアルタイム：** ActionCable (WebSocket)
- **デプロイ先：** Render.com（Web Service + PostgreSQL）
- **使用予定ライブラリ：**
  - `inertia_rails`（Rails↔React橋渡し）
  - `vite_ruby` + `@vitejs/plugin-react`（フロントエンドビルド）
  - Tailwind CSS v4
  - `@rails/actioncable`（WebSocketクライアント）

### 10-2. 技術選定の理由

- **Rails：** 業務ロジック・DBまわりを素早く実装でき、ActionCableが標準搭載のためリアルタイム機能を追加コストなく実現できる
- **Inertia.js：** APIを別途設計せずRailsのコントローラーからReactコンポーネントに直接データを渡せるため、小規模アプリに適している
- **React + SVG直書き：** 外部グラフィクスライブラリに依存せず、座席・壁をSVG要素として直接操作することでドラッグ処理を完全にコントロールできる
- **チャレンジしたい点：** ActionCableとReact状態管理の組み合わせ、SVGのインタラクション実装（以前の出席管理アプリをより直感的な操作に進化させる）
- **不安な点：** SVGドラッグ操作のタッチ対応、ActionCable接続管理の本番環境での安定性

### 10-3. キャッチアップ不足の懸念

Inertia.jsはRailsともReactとも相性が良いが、独自のデータフロー（Inertia Pageオブジェクト）の理解に初期コストがかかる。
SVGのドラッグ実装は標準的なReactイベント処理とは異なる座標計算が必要なため、まずシンプルな単一座席の移動から実装して動作確認し、段階的に機能を増やす進め方を取る。
ReactおよびActionCableについては、本アプリへの組み込み前にそれぞれ単独で動作確認するスパイクを設ける。
なお、以前に出席管理アプリを開発した経験があるため、入退室の記録・状態管理の設計については一定の知見がある。その経験を土台にしつつ、SVGエディタとリアルタイム同期という新しい技術要素に集中してキャッチアップする。
